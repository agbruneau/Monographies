# ADR-0005 — SDK A2A : a2aproject/a2a-go v2.x

- **Statut :** accepté
- **Date :** 2026-07-07
- **Phase :** M2.8

## Contexte

Le PRD supposait `a2a-go` en **v1.x**. P0.1 a constaté que le dépôt canonique
`github.com/a2aproject/a2a-go` est en **v2.3.1** (module `/v2`, Go ≥ 1.24.4,
Apache 2.0). Le SDK fournit les types (`AgentCard`, `Task`, `Message`,
`TaskState` — 8 états), le modèle d'exécuteur (`iter.Seq2[Event, error]`), les
handlers serveur (`NewHandler`, `NewJSONRPCHandler`, `NewStaticAgentCardHandler`),
le client (`NewFromCard`, `SendMessage`, `SendStreamingMessage`) et le resolver
de cards (`agentcard.DefaultResolver`). Il ne fournit **pas** de signature de
card (JWS/JCS) : celle-ci est implémentée en stdlib (`pkg/a2a`, ADR-0001).

## Décision

Dépendre de **`github.com/a2aproject/a2a-go/v2 v2.3.1`** (chemin `/v2`). Les
agents sont des serveurs A2A (`internal/a2aserver`) exposant leur Agent Card
signée et un point JSON-RPC ; l'orchestrateur est un client A2A
(`internal/orchestrator`) découvrant par card, vérifiant les signatures et
routant par compétence. Le cycle de vie de Task est piloté par l'exécuteur
(`NewSubmittedTask` → `NewStatusUpdateEvent`).

## Conséquences

- **Écart au PRD acté** (v1 → v2) : chemin de module `/v2`, API par exécuteur
  itératif (Go 1.23+). Sans impact sur les exigences (les capacités A2A
  requises sont toutes présentes).
- Signature de card à notre charge (stdlib, `pkg/a2a`) — maîtrisée et testée.
- Transport JSON-RPC retenu (ADR-0004) ; gRPC/REST disponibles mais non requis.

## Reversal condition

Un abandon du dépôt `a2aproject/a2a-go` ou une rupture d'API v3 sans chemin de
migration justifierait de réévaluer (autre SDK, ou implémentation directe du
sous-ensemble A2A requis).

## Alternatives écartées

- **Implémenter A2A à la main** : coûteux (HTTP + JSON-RPC + SSE + cycle de vie
  de Task + resolver), contraire à *ne pas réinventer*.
- **`a2a-go` v1** : n'existe pas comme voie active (P0.1).
